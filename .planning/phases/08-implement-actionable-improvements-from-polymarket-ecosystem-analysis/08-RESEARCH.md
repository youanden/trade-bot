## Summary

Phase 8 implements the four highest-priority improvements identified in the polymarket ecosystem analysis (quick task 260322-f6e, ANALYSIS.md). The analysis compared 6 Polymarket open-source repos against our codebase and produced a prioritized gap list. This research translates those gaps into specific, implementation-ready findings.

The work is entirely in the exchange client layer (`src/worker/core/exchanges/polymarket/`), the database schema (`src/worker/core/db/schema.ts`), a new Drizzle migration, and new unit tests. No strategy code is modified. No architecture changes â€” this is targeted correctness and resilience work.

**Critical finding:** Our `clobFetch` HMAC signing has two bugs relative to the official Polymarket `clob-client` SDK â€” the secret is not base64-decoded before use, and the output is standard base64 instead of URL-safe base64. These bugs cause authenticated requests to silently fail or produce incorrect signatures in production. This is the highest-priority fix.

**Primary recommendation:** Fix the HMAC auth bug first (it breaks all authenticated endpoints in production), then add neg-risk exchange detection (prevents order failures on ~30% of Polymarket markets), then add structured HTTP error handling and retry logic (resilience), then add `clobTokenIds` and `negRiskMarketId` to the schema (correctness for order routing).

### Pattern 1: HMAC Fix â€” Base64-Decoded Secret + URL-Safe Output

**What:** Polymarket's official `clob-client` SDK (`/src/signing/hmac.ts`) imports the `apiSecret` by first decoding it from base64, then uses URL-safe base64 output encoding.

**Current bug in `clobFetch`:**
```typescript
// WRONG â€” treats secret as raw UTF-8 text:
const key = await crypto.subtle.importKey(
  "raw",
  new TextEncoder().encode(this.config.apiSecret),  // BUG: should be base64-decoded
  { name: "HMAC", hash: "SHA-256" },
  false,
  ["sign"]
);
const sigB64 = btoa(String.fromCharCode(...new Uint8Array(sig)));  // BUG: not URL-safe
```

**Correct pattern (matching official SDK):**
```typescript
// Source: github.com/Polymarket/clob-client/blob/main/src/signing/hmac.ts
// Step 1: base64-decode the secret (handle both standard and URL-safe base64)
function base64ToArrayBuffer(b64: string): ArrayBuffer {
  const normalized = b64.replace(/-/g, "+").replace(/_/g, "/");
  const binary = atob(normalized);
  const bytes = new Uint8Array(binary.length);
  for (let i = 0; i < binary.length; i++) {
    bytes[i] = binary.charCodeAt(i);
  }
  return bytes.buffer;
}

const key = await crypto.subtle.importKey(
  "raw",
  base64ToArrayBuffer(this.config.apiSecret),  // FIXED: decode from base64
  { name: "HMAC", hash: "SHA-256" },
  false,
  ["sign"]
);

const sig = await crypto.subtle.sign("HMAC", key, new TextEncoder().encode(message));

// Step 2: URL-safe base64 output
const sigB64 = btoa(String.fromCharCode(...new Uint8Array(sig)))
  .replace(/\+/g, "-")
  .replace(/\//g, "_");  // FIXED: URL-safe encoding
```

**Confidence:** HIGH â€” verified against official SDK source and Polymarket docs.

**Note on timestamp:** Our code uses `Math.floor(Date.now() / 1000)` which gives Unix seconds. This is correct for L2 CLOB auth (seconds, not milliseconds). No change needed here.

### Pattern 2: Neg-Risk Exchange Detection

**What:** Polymarket uses two separate CTF Exchange contract addresses. Orders signed against the wrong contract fail on-chain. Both addresses are already defined in our `POLY_CONTRACTS` constants.

- Standard CTF Exchange: `0x4bFb41d5B3570DeFd03C39a9A4D8dE6Bd8B8982E`
- Neg Risk CTF Exchange: `0xC5d563A36AE78145C45a50134d48A1215220f80a`

The `negRisk` flag is already present in our `GammaMarket` and `ClobMarket` types. The `gammaToMarketInfo` mapper already stores `negRisk` in `meta`. The problem is `placeOrder` ignores it.

**Where the fix lives:** `placeOrder` receives an `OrderRequest` whose `marketId` is a token ID string. We need `isNegRisk` to reach `placeOrder`. Options:

**Option A â€” Extend `OrderRequest.meta`:** Add `negRisk?: boolean` to `OrderRequest` type. Callers (strategies) pass it via market metadata. LOW effort.

**Option B â€” Store negRisk in DB schema column, look up in client:** Add `negRiskMarketId` to `markets` table (already planned for schema additions). Client queries before signing. MEDIUM effort, requires DB access in client.

**Option C â€” Accept negRisk flag in constructor or per-call:** Add optional `isNegRisk?: boolean` to `OrderRequest`. LOWEST effort, enables strategy control.

**Recommendation:** Option C â€” add `isNegRisk?: boolean` to `OrderRequest` in `types.ts`. Strategies that care set it explicitly; default false (standard exchange) preserves backward compatibility.

```typescript
// In placeOrder after the TODO comment:
const exchangeAddress = order.isNegRisk
  ? POLY_CONTRACTS.NEG_RISK_EXCHANGE
  : POLY_CONTRACTS.CTF_EXCHANGE;
```

**Schema addition (separate concern):** Add `negRiskMarketId text` column to `markets` table so market sync can store it, but this is advisory data â€” the `OrderRequest.isNegRisk` flag is the actual routing mechanism.

**Confidence:** HIGH â€” contract addresses verified on PolygonScan, flag already in our type definitions.

### Pattern 3: Structured HTTP Error Handling

**What:** Currently `clobFetch` throws a generic string error on non-2xx responses. This prevents callers from distinguishing retryable (429, 503) from permanent (400, 401) failures. Polymarket's builder SDK extracts `{ status, statusText, data }` and uses typed error classes.

**Pattern:**
```typescript
// In clobFetch, replace the current throw with structured extraction:
if (!res.ok) {
  let body = "";
  try { body = await res.text(); } catch {}
  const err = new ClobApiError(res.status, `CLOB ${method} ${path}`, body);
  throw err;
}

// New class in polymarket/types.ts:
export class ClobApiError extends Error {
  constructor(
    public readonly status: number,
    public readonly context: string,
    public readonly body: string
  ) {
    super(`${context} ${status}: ${body}`);
    this.name = "ClobApiError";
  }
  get isRetryable(): boolean {
    return this.status === 429 || this.status >= 500;
  }
  get isAuthError(): boolean {
    return this.status === 401 || this.status === 403;
  }
}
```

**Confidence:** HIGH â€” pattern derived from Polymarket builder SDK analysis + standard HTTP retry conventions.

### Pattern 4: Retry with Exponential Backoff

**What:** Network errors and 5xx/429 responses should be retried for idempotent operations (GET requests). POST order placement should NOT retry on API-level rejections (risk of duplicate orders).

**Pattern:**
```typescript
// Wrap clobFetch calls in a retry utility:
async function withRetry<T>(
  fn: () => Promise<T>,
  maxAttempts = 3,
  baseDelayMs = 500
): Promise<T> {
  for (let attempt = 0; attempt < maxAttempts; attempt++) {
    try {
      return await fn();
    } catch (err) {
      const isLast = attempt === maxAttempts - 1;
      if (isLast) throw err;
      if (err instanceof ClobApiError && !err.isRetryable) throw err;
      const delay = baseDelayMs * Math.pow(2, attempt);
      await new Promise(r => setTimeout(r, delay));
    }
  }
  throw new Error("unreachable");
}
```

Apply retry to: `getMarkets`, `getMarket`, `getPrice`, `getOrderBook`, `getOrder`, `getOpenOrders`, `getPositions`, `getBalance`.
Do NOT apply retry to: `placeOrder`, `cancelOrder` (non-idempotent).

**CF Workers constraint:** `setTimeout` is available in Cloudflare Workers (not truly async in the same way, but acceptable for short delays). Alternatively, use `scheduler.wait()` if needed. SHORT delays (< 1s per retry) are fine.

**Confidence:** MEDIUM â€” standard retry pattern, CF Workers compatibility with setTimeout verified via CLAUDE.md constraints noting `nodejs_compat` is enabled.

## Common Pitfalls

### Pitfall 1: Secret Encoding Mismatch
**What goes wrong:** The `apiSecret` stored in Cloudflare secrets is already base64-encoded (Polymarket issues it as a base64 string). If you pass it raw to `TextEncoder`, the HMAC key material is wrong, producing signatures that look valid but are rejected by the CLOB API.
**Why it happens:** Not obvious from the field name `apiSecret` that it's base64-encoded.
**How to avoid:** Always `atob(secret)` before using as HMAC key material. Add a comment in code.
**Warning signs:** 401 responses on authenticated endpoints with a correct API key, passphrase, and timestamp.

### Pitfall 2: URL-Safe vs Standard Base64 in Signature
**What goes wrong:** Standard `btoa()` output contains `+` and `/` which are illegal in some HTTP header values and cause signature mismatch on the server.
**Why it happens:** Easy to forget â€” `btoa()` is the obvious "base64 encode" function.
**How to avoid:** Always apply `.replace(/\+/g, '-').replace(/\//g, '_')` after `btoa()` for HMAC output.
**Warning signs:** 401 on requests where the signature contains `+` or `/` characters.

### Pitfall 3: Retrying Non-Idempotent POST /order
**What goes wrong:** If `placeOrder` is retried on a 5xx response, the first order may have been partially processed, resulting in duplicate orders on the exchange.
**Why it happens:** Applying a blanket retry wrapper to all methods.
**How to avoid:** Keep `placeOrder` and `cancelOrder` outside the retry wrapper. Only retry GET-equivalent methods.
**Warning signs:** Duplicate open orders visible in the exchange UI.

### Pitfall 4: neg-risk detection via token ID prefix is unreliable
**What goes wrong:** Some implementations try to detect neg-risk markets by inspecting the token ID numeric range. This is not a documented guarantee.
**Why it happens:** Trying to avoid passing `isNegRisk` from caller to client.
**How to avoid:** Use the `negRisk` boolean from the Gamma API response. Store it in market metadata or pass as `OrderRequest.isNegRisk`. Trust the API field, not token ID heuristics.
**Warning signs:** Orders on neg-risk markets failing with contract-level errors.

### Pitfall 5: Schema Column Addition Breaks In-Memory Test DB
**What goes wrong:** Test helpers that apply the schema to in-memory SQLite via `drizzle-kit push` or raw SQL may not pick up new columns added to `schema.ts` if the migration file isn't applied.
**Why it happens:** Test DB setup uses `schema.ts` directly via Drizzle, not migration SQL files.
**How to avoid:** After adding columns to `schema.ts`, verify `TestDb` (in `test/helpers/`) still applies schema correctly. Our test setup uses `createTestDb()` which should apply the schema automatically since it uses the Drizzle schema object, not migration files.
**Warning signs:** TypeScript compile errors on schema tests, or "no such column" SQLite errors in tests.

## Open Questions

1. **`OrderRequest` interface is shared with Kalshi**
   - What we know: Adding `isNegRisk?: boolean` to `OrderRequest` in `types.ts` adds a Polymarket-only field to the shared interface
   - What's unclear: Whether this is better than a Polymarket-specific extension type
   - Recommendation: Add it as `isNegRisk?: boolean` â€” it's an optional field, Kalshi client ignores it, and the interface stays unified

2. **Should `OrderRequest.isNegRisk` come from the strategy or from DB lookup?**
   - What we know: Strategies currently don't pass `isNegRisk` â€” they pass `marketId` (token ID). The market's `negRisk` flag lives in `meta.negRisk` from market discovery
   - What's unclear: Whether strategies have the market metadata available when calling `placeOrder`
   - Recommendation: For Phase 8, strategies that currently have market data available (cross-arb, market-maker) can populate the field. The field defaults to `false` which is backward-safe. A follow-on improvement can add DB lookup in the client. Do NOT add DB access to the exchange client â€” it doesn't have a DB reference today.

3. **Drizzle migration and test DB schema sync**
   - What we know: `createTestDb()` in test helpers applies `schema.ts` via Drizzle ORM â€” it picks up new columns automatically because it references the TypeScript schema object
   - What's unclear: Whether the `schema.test.ts` test verifies all columns explicitly (it should be checked)
   - Recommendation: Review `test/core/schema.test.ts` before adding columns; ensure it doesn't hard-code column lists that would need updating
